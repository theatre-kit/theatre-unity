### 3. Move SpatialResultBuilder to Spatial/ [DONE]

**Rule:** shared-utility-subfolder (partial)
**What was done:** `SpatialResultBuilder.cs` moved from Editor/Tools/ root into
Editor/Tools/Spatial/ where its only 2 callers live. Namespace updated to
`Theatre.Editor.Tools.Spatial`.

**What was NOT done:** `JsonParamParser.cs` and `ResponseHelpers.cs` were NOT moved to a
`Runtime/Stage/Shared/` subfolder. During implementation, we discovered `ResponseHelpers`
is referenced by **30 files** across both Runtime and Editor assemblies — not the ~15
estimated in the original backlog. Moving it to `Theatre.Stage.Shared` would require adding
a `using` statement to 30 files. This is the same "too much churn for cosmetic benefit"
reasoning that puts the Runtime/Stage namespace split in "Not Worth It."

## Worth Considering

### 5. Align Runtime/Stage/ sub-namespace to folders

**Rule:** namespace-folder-alignment
**Current:** All files in `Runtime/Stage/GameObject/` and `Runtime/Stage/Spatial/` use
the flat `namespace Theatre.Stage`.

| Folder | Files | Current Namespace | Aligned Namespace |
|--------|-------|-------------------|-------------------|
| Stage/GameObject/ | 6 files | `Theatre.Stage` | `Theatre.Stage.GameObject` |
| Stage/Spatial/ | 6 files | `Theatre.Stage` | `Theatre.Stage.Spatial` |

**Effort:** Moderate. 12 files need namespace changes. Callers across Editor/Tools/ (all
tools that `using Theatre.Stage`) would need updated imports. Some types are referenced
extensively (e.g., `HierarchyWalker`, `ObjectResolver`, `TokenBudget`).

**Trade-off:** Better discoverability vs. more `using` statements in tool files. Since
Runtime/Stage/ only has 14 files total, the flat namespace isn't causing real confusion yet.

## Not Worth It

### Runtime/Stage/GameObject/ and Runtime/Stage/Spatial/ namespace change (now)

**Rule violated:** namespace-folder-alignment (item 5 above)
**Why skip:** The flat `Theatre.Stage` namespace is used by virtually every Editor tool file.
Splitting into `Theatre.Stage.GameObject` and `Theatre.Stage.Spatial` would require adding
2 new `using` statements to ~20 files. The 14 files in Stage/ are not causing navigation
confusion at current scale. Revisit when Stage/ exceeds 20 files or when a new subsystem
(e.g., ECS) adds ambiguity.

### Moving JsonParamParser/ResponseHelpers to Shared/ subfolder

**Rule violated:** shared-utility-subfolder
**Why skip:** `ResponseHelpers` is referenced by 30 files across both Runtime and Editor
assemblies. Moving to `Theatre.Stage.Shared` would add a `using` to all 30 files for a
purely cosmetic folder change. The 2 files sitting at `Runtime/Stage/` root is not causing
real confusion — they're clearly named as utilities. Revisit if Stage/ root grows past
5+ loose files.

### Splitting any current files for size

**Rule violated:** file-size-cap (500 LOC)
**Why skip:** No file exceeds the 500-line cap. The largest file is `SceneDeltaTool.cs` at
392 LOC — well within limits. No action needed.

### Moving test files to match source folder grouping

**Rule violated:** feature-grouped-tests
**Why skip:** Current test grouping is already feature-based and working well:
- `SceneAwarenessTests` covers hierarchy/resolver/response helpers
- `WatchActionTests` covers watch engine/conditions/actions
- `SpatialQueryTests` covers all spatial queries
- `McpIntegrationTests` covers transport layer

No test file exceeds 310 LOC. The grouping is natural and doesn't need restructuring.

### Removing Director/ placeholder folders

**Rule:** keep-placeholders
**Why skip:** Explicitly kept per team preference. The 7 empty subdirectories
(Scenes, Prefabs, Assets, Animation, Spatial, Input, Config) document the Phase 5+
roadmap visible in the file tree.

### Restructuring docs/

**Rule:** docs-layout
**Why skip:** Current layout (foundation docs at `docs/` root, phase designs in
`docs/designs/`) is clean and follows the agreed pattern. No changes needed.
